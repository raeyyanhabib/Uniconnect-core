SOLID Principles in UniConnect

This file explains how we used SOLID principles in the website.


1. Single Responsibility Principle (SRP)
Each part of the code should only do one thing.
Where it's used:
- api.ts only handles network requests and login tokens.
- theme.ts only handles colors and styles.
- Components like Avatar and Modal just handle their own specific UI.
Improvement: App.tsx is a bit big right now because it handles routing, auth, and layout all at once. We could split it up later.


2. Open-Closed Principle (OCP)
The code should be easy to add onto without breaking what's already there.
Where it's used:
- navGroups in App.tsx makes it easy to add new pages just by adding to the list.
- theme.ts uses color tokens so we can change the whole look without touching every file.


3. Liskov Substitution Principle (LSP)
Different components should be able to swap out if they follow the same rules.
Where it's used:
- React components use standard props, so they are easy to swap.
- UserType is used everywhere so any page expecting a user gets exactly what it needs.


4. Interface Segregation Principle (ISP)
Don't force a component to use data it doesn't need.
Where it's used:
- AvatarProps only asks for a name and size, not the whole user profile.
- Sidebar and TopBar have their own specific data requirements.


5. Dependency Inversion Principle (DIP)
High level code shouldn't depend on low level details.
Where it's used:
- The api object hides the fetch details from the pages.
- Components use theme tokens instead of hardcoded hex codes.
