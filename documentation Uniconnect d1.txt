# UniConnect Project Documentation & Master Log

## 1. Project Origins & The Core Vision
UniConnect originated from a master blueprint (`democore.txt`) designed to build the ultimate, highly-secure university ecosystem. It was conceived as a monolith React-Node application bridging academic networking (Study Partners, Groups), an active campus marketplace (Resource Lending), secure communications (Messages), and administrative oversight.

The project is actively being hydrated—transitioning from its initial mockup state powered by massive static React arrays into a fully networked, SQLite-driven, REST-fueled powerhouse.

## 3. Database Schema Overview (SQLite)

The application relies on `better-sqlite3` initialized via `server/db.js`. 
- `Users`: Primary tracking entity. Contains sensitive auth data (hashed via `bcrypt`), JWT tokens, roles (`student` vs `admin`), profile data, and privacy toggles.
- `PartnerRequests`: Joins `fromId` and `toId` tracking the `pending`, `accepted`, or `declined` states.
- `StudyGroups`: Contains maximum capacities, course codes, and visibility.
- `GroupMembers`: Relational mapping tracking which User belongs to which Group, and their specific authorization `role`.
- `GroupAnnouncements`: Bound uniquely to study groups ensuring only group members see broadcasts.
- `Resources`: Defines active marketplace items, statuses, classifications, and `reportCount` for Admin flagging flags.
- `Transactions`: Ledger of every borrow event tying a `resourceId`, an `ownerId`, and a `borrowerId` together through complex handover status enums indefinitely tracking the lifecycle of an asset.
- `Messages` & `Notifications`: Chat structures mapped by timestamps and `read` boolean flags.
- `LostAndFound` & `Reports`: Auxiliary support tables scaling admin oversight.

## 4. File Architecture & Data Flow
The project utilizes a Mono-repo layout tying a React/Vite front to an Express back.

### The Frontend Layer (`/src`)

- `main.tsx`: Bootstraps the React DOM and triggers Vite HMR.
- `api.ts`: A critical Axios/Fetch proxy intercepting all requests, verifying `localStorage` tokens, and injecting generic `Authorization: Bearer <token>` Headers.
- `App.tsx`: The colossal monolith component housing all logical views (`<LoginPage>`, `<DashboardPage>`, `<StudyPartnersPage>`, `<AdminDashboardPage>`, etc). Driven purely by React state routing.
- `index.css`: Supplies globally accessible tokens, glass-morphism classes, and UI animations.

### The Backend Layer (`/server`)

- `index.js`: Core server bootstrap. Implements CORS payloads, parses incoming `json`, and routes endpoints.
- `db.js`: Reconstructs database schemas aggressively if they don't exist.
- `middleware/auth.js`: Supplies `authMiddleware` (validating JWT integrity) and `adminMiddleware` (validating account `role` authorization levels). Now requires a proper `JWT_SECRET` from `.env` — the hardcoded fallback has been removed to prevent token forgery.

- `routes/`: Distinct API routers isolating logic:
    - auth.js (UC 1-3) — Now enforces FAST-NU email validation on registration and blocks unverified users from logging in
    - users.js (UC 4-6, Aggregations)
    - partners.js (UC 7-9)
    - groups.js (UC 10-14)
    - resources.js (UC 15-27)
    - messages.js, notifications.js, lostfound.js
    - admin.js (UC 32-34)

## 5. Security Hardening (Completed — AbdulRehman)

The following critical security vulnerabilities were identified and resolved:

1. **Hardcoded Credentials Removed**: Login page no longer pre-fills test credentials (`alex.chen@university.edu` / `password123`) or hardcodes admin credentials (`admin@university.edu` / `admin123`) in the JavaScript bundle.

2. **Secrets Excluded from Git**: Added `server/.env`, `*.db`, `*.db-shm`, `*.db-wal` to `.gitignore`. Removed previously tracked `.env` and SQLite database files from version control. Created `server/.env.example` as a safe configuration template.

3. **JWT Fallback Eliminated**: The auth middleware previously fell back to a known secret (`uniconnect-dev-secret`) if `JWT_SECRET` wasn't set. The server now refuses to start without a properly configured `.env` file.

4. **Verification Enforcement on Login**: Unverified students are now blocked from logging in with a 403 error until they complete the verification step (UC 2/3). Admin accounts are exempt from this check.

5. **Server-Side Email Domain Validation**: Registration now validates the FAST-NU email format (`l240690@lhr.nu.edu.pk`) on the backend, preventing attackers from bypassing the frontend validation with curl or Postman to register with non-university emails.
