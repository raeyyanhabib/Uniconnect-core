The Technology Stack Fundamentals

Every tool, language, and framework used in UniConnect — what it is, what it does, and why we chose it.

⚛

React

UI Library

React is a JavaScript library created by Facebook for building user interfaces. Instead of reloading entire pages, React builds your app out of small, reusable "components" — self-contained blocks of UI (like a button, a card, or an entire page). When data changes, React automatically figures out the minimum number of updates needed and re-renders only the affected parts. This makes apps feel instant and responsive.

Why we use it: It lets us build a complex 17-page app from modular pieces. Each page is a separate file, each button a reusable component.
TS

TypeScript

Language

TypeScript is JavaScript with types. In plain JavaScript, a variable can hold anything — a number, a string, an object — and the language won't warn you if something is wrong until the code actually runs and crashes. TypeScript adds a layer where you define what shape your data should have (e.g., "a User always has a name and an email"). If you make a mistake, the editor catches it instantly before you even run the code.

Why we use it: With 30+ files exchanging data, TypeScript prevents bugs by ensuring every component knows exactly what data it will receive.
⚡

Vite

Build Tool & Dev Server

Vite (French for "fast") is the tool that runs your React project during development. When you type npm run dev, Vite starts a local web server, watches your files, and instantly refreshes the browser when you save a change — a feature called Hot Module Replacement (HMR). For production, Vite bundles all your files into optimized, minified assets. Think of it as the engine that converts your source code into something a browser can actually understand.

Why we use it: It's the fastest dev server available. Changes reflect in under 100ms. The older alternative (Create React App) could take 10-30 seconds.
Ex

Express.js

Backend Framework

Express is a minimal Node.js framework for building web servers. It handles incoming HTTP requests (GET, POST, PUT, DELETE) and routes them to the right handler function. When the React frontend needs data — like "fetch all study groups" — it sends a request to the Express server, which queries the database, processes the result, and sends back JSON data. Express is the bridge between the UI and the database.

Why we use it: It's the most battle-tested Node.js server framework in existence. Simple to learn, powerful enough for production.
🗄

SQLite

Database

SQLite is a relational database that stores all its data in a single file on disk (uniconnect.db). Unlike MySQL or PostgreSQL which require running a separate server process, SQLite runs directly inside your Node.js application. It supports full SQL — tables, joins, foreign keys, indexes — but with zero configuration. You literally just point it at a file and start querying. It's used in billions of devices worldwide (every iPhone has SQLite).

Why we use it: Zero setup. No database server to install. Perfect for a university project that needs to be portable and easy to demo.
🌐

Node.js

Runtime Environment

Node.js lets you run JavaScript outside of a web browser. Normally, JavaScript only works inside Chrome, Firefox, etc. Node.js took Chrome's V8 engine and packaged it as a standalone program, so you can use JavaScript to build servers, command-line tools, and more. Our Express backend runs on Node.js. When you type npm start, you're telling Node.js to execute the server code.

Why we use it: One language (JavaScript/TypeScript) for both frontend AND backend. No context-switching between languages.
🔑

JWT (JSON Web Tokens)

Authentication System

JWT is how we handle login sessions. When a user logs in, the server creates a signed token — an encoded string containing the user's ID and role. This token is sent back to the browser and stored in localStorage. Every subsequent API request includes this token in the headers. The server decodes it to identify who's making the request, without needing to store session data. If the token is tampered with, the signature check fails and access is denied.

Why we use it: Stateless authentication. The server doesn't need to remember who's logged in — the token itself contains all necessary info.
📦

npm (Node Package Manager)

Package Manager

npm is the tool that manages external libraries (called "packages" or "dependencies"). Instead of writing everything from scratch, you install pre-built solutions — like bcryptjs for password hashing. The package.json file lists every dependency, and npm install downloads them all into a node_modules folder. Think of npm as an app store for code libraries.

Why we use it: It comes with Node.js by default and hosts over 2 million packages. Every JavaScript project uses it.

Step by step: You click "Sign In" → React calls api.post('/api/auth/login', ...) → Vite sees /api and forwards the request to the Express server at port 3001 → Express routes it to routes/auth.js → The handler queries the Users table in SQLite → Verifies the password hash → Creates a JWT token → Sends it back as JSON → React stores the token and navigates to the Dashboard.
🔑
The Proxy trick: The React frontend runs on port 5174, but the backend runs on port 3001. Normally, browsers block requests across different ports (CORS). The vite.config.ts says: "Any request starting with /api — secretly forward it to port 3001." So the browser thinks it's talking to one server, but Vite is silently routing API calls to Express behind the scenes.

Frontend Files Breakdown src/

Every single file in the frontend, what it does, and how it fits into the bigger picture.

Entry Points

main.tsx Entry

The very first file that runs. It imports React, finds the <div id="root"> element in index.html, and mounts the entire <App /> component inside it. This is the "ignition key" of the application. It also imports index.css so global styles load immediately. Wrapped in <StrictMode> which enables extra development warnings.

App.tsx Master Router

The central orchestrator. Manages three critical pieces of state: currentUser (who's logged in), currentPage (which page to show), and isAdmin (student vs admin mode). It renders the AppShell layout (Sidebar + TopBar) and conditionally renders the correct page component based on currentPage. Also handles session restoration from localStorage on page reload, login/logout flows, and the navigation function passed down to every page.

index.css Global Styles

Contains all global CSS rules — the dark background, font imports (Inter from Google Fonts), glassmorphism card effects (.authCard), keyframe animations (@keyframes fadeIn, @keyframes slideUp), message bubble styles, hover transitions, and CSS variables. Every component in the app inherits from these base styles.

Services — src/services/

api.ts Service

The HTTP client for the entire app. Exports an api object with methods: api.get(), api.post(), api.put(), api.patch(), api.delete(). Each method calls a generic request() function that automatically attaches the JWT token from localStorage as a Bearer header, converts the body to JSON, and throws descriptive errors on failure. Also exports setToken() and getToken() for managing login state.

theme.ts Service

The design token file. Exports a C (Colors) object containing every color used in the app — background shades (base, depth, surface), accent colors (cyan, amber, green, red, purple), and text colors (tx, txS, txM). Also exports pre-built style objects: inp (input field), btnP (primary button), btnS (secondary), btnD (danger/red), btnG (green/success), and cardStyle() (standard card wrapper). Changing a color here updates it everywhere.

Types — src/types/

index.ts Type Definitions

Defines the TypeScript interfaces (data shapes) used across the entire app. User describes a student/admin account (userId, name, email, department, semester, status, role, etc.). Resource describes a marketplace item. StudyGroup describes a group with members and announcements. AuthPageProps defines the props that authentication pages receive (navigation and login callbacks). These ensure every component gets exactly the data it expects.

Shared Components — src/components/

Avatar.tsx Component

Renders a circular avatar with the user's initials. Takes name and size as props. Generates a deterministic background color from the name string using a hash function, so the same name always produces the same color. Used throughout the app wherever a user's face would normally appear.

Badge.tsx Component

A small colored label used to tag entities. Accepts a label string and a color. Renders as a pill-shaped element with a semi-transparent background tinted to the given color. Used for things like "CS401", "Creator", "Book", etc.

StatusBadge.tsx Component

A specialized badge that maps statuses to colors and labels. Pass status="available" and it renders a green "Available" pill. Pass status="blocked" and it renders red. Handles: available, active, borrowed, pending, paused, flagged, public, private, resolved, blocked, rejected, approved.

StarRating.tsx Component

Renders 1-5 stars. Filled stars are amber, empty stars are grey. Takes a rating number. Used in transaction history and user reviews to display borrower feedback scores.

StatCard.tsx Component

A dashboard metric card showing a label, large value, icon, and subtitle. Used on the Dashboard and Admin Dashboard to display stats like "Active Users: 42" or "Pending Reports: 3".

Tabs.tsx Component

A horizontal tab bar. Takes an array of tab objects (id, label, optional count badge) and renders clickable pills. The active tab gets a highlighted style. Used in Study Groups ("My Groups / Discover / Create"), Resources ("Browse / Post / My Listings"), etc.

SectionHeader.tsx Component

A page title block with a title, subtitle, and optional action button on the right. Used at the top of every major page to provide a consistent header layout.

Modal.tsx Component

A full-screen overlay with a centered card. Shows a title, close button (X), and renders any children inside. Used for forms like "Schedule Handover", "Rate User", and "Report Item". Clicking the backdrop closes it.

FormField.tsx Component

A simple wrapper that renders a label above any form input. Takes a label string and wraps its children with consistent spacing and typography. Keeps all forms looking uniform.

EmptyState.tsx Component

Shown when a list has no items. Displays a large icon, title text, optional subtitle, and optional action button. Example: "You haven't joined any groups" with an "Explore Groups" button.

StudentCard.tsx Component

A card displaying a student's info — avatar, name, department, mutual courses, and action buttons (Send Request, View Profile). Used on the Study Partners page to browse potential partners.

ResourceCard.tsx Component

A card showing a marketplace resource — title, category badge, condition, status, owner name, and custom action elements (Borrow, Contact). Used on the Resources browse tab.

Pages — src/pages/

📁
Each page is a self-contained React component managing its own state, API calls, and rendering. Pages receive user and onNavigate props from App.tsx.
LoginPage.tsx Auth Page

The student login screen. Renders email/password fields with the UniConnect branding. On submit, calls POST /api/auth/login, stores the returned JWT token, and passes the user data up to App.tsx via onLogin(). Also has links to Register, Forgot Password, and Admin Portal.

AdminLoginPage.tsx Auth Page

Identical flow to LoginPage but styled differently with a shield icon and admin branding. Authenticates via the same /api/auth/login endpoint but sends admin credentials.

RegisterPage.tsx Auth Page

New user registration form. Collects name, email (must end in @university.edu), password (with confirmation), department, and semester. Calls POST /api/auth/register. On success, navigates to the verification page.

VerifyStudentPage.tsx Auth Page

Post-registration verification screen. Accepts a Student ID for identity confirmation. Calls POST /api/auth/verify. On success, navigates to login.

ResetPasswordPage.tsx Auth Page

Password recovery form. Takes email + Student ID to verify identity, then accepts a new password. Calls POST /api/auth/reset-password.

DashboardPage.tsx App Page

The main landing screen after login. Fetches GET /api/users/dashboard which returns aggregated stats (partner count, group count, resource count). Renders StatCards and a welcome message with the user's name. Quick-action buttons navigate to other pages.

ProfilePage.tsx App Page

Displays and edits the current user's profile. Shows avatar, name, department, bio, semester, average rating. Has an edit mode toggled by a button. Fetches GET /api/users/profile and saves via PUT /api/users/profile. Also contains privacy toggles (show email, show rating, allow partner requests) that send PUT /api/users/privacy.

StudyPartnersPage.tsx App Page

Three-tab layout: Discover (browse students), My Partners (accepted connections), Requests (pending inbound/outbound). Fetches from GET /api/partners, GET /api/partners/requests. Search and filter by name/department. Send requests via POST /api/partners/request, accept/decline via PUT /api/partners/:id.

StudyGroupsPage.tsx App Page

Three tabs: My Groups, Discover, Create Group. Group detail view shows members list (with remove capability for creators), announcement feed, and leave button. Create form has name, description, course code, max members, and public/private toggle. APIs: /api/groups/mine, /api/groups/discover, /api/groups/:id, POST /api/groups, etc.

ResourcesPage.tsx App Page

The largest page (~17KB). Five tabs: Browse (searchable catalog with category filters), Post Resource (listing form), My Listings (with pause/resume/delete), Borrow Requests (approve/reject inbound requests), and Lending History (with return confirmation and rating). Modals for scheduling handovers, rating borrowers, and filing disputes.

MessagesPage.tsx App Page

Split-pane messaging interface. Left sidebar shows conversation list (with online indicators and unread badges) and suggested study partners. Right pane displays the active conversation with message bubbles (styled differently for sent vs received). Real-time polling every 2 seconds. APIs: GET /api/messages, GET /api/messages/:id/messages, POST /api/messages/send.

NotificationsPage.tsx App Page

Chronological feed of system notifications. Each notification has a type-specific icon (partner request = UserPlus, borrow = Package, group = Layers) and color. Click to mark as read. "Mark all read" button. Polls GET /api/notifications every 3 seconds.

LostFoundPage.tsx App Page

Campus lost & found board. Cards show item type (Lost/Found), description, location, reporter, and status. Filter buttons: All, Lost, Found, Open. Report modal collects description, location, and optional image. APIs: GET /api/lost-found, POST /api/lost-found, PUT /api/lost-found/:id/resolve.

AdminDashboardPage.tsx Admin Page

Admin-only overview. Shows platform-wide stats (total users, active loans, overdue items, pending reports). Lists pending actions with links to relevant admin pages. Shows platform health bars. Fetches GET /api/admin/dashboard.

AdminUsersPage.tsx Admin Page

Full user management table. Columns: User (avatar + name + email), Department, Status (active/blocked), Joined date, Reports, Actions. Search bar filters by name or email. Block/Unblock toggle calls PUT /api/admin/users/:id/block.

AdminResourcesPage.tsx Admin Page

Moderates marketplace listings. Shows each resource with owner info, category, status, and report count. Flagged items get a red border. Remove button calls DELETE /api/admin/resources/:id.

AdminReportsPage.tsx Admin Page

Displays user-submitted reports (abuse, harassment, disputes). Each report shows type, description, reported user, reporter, date, and status. Admin actions: Start Investigation, Mark Resolved, Escalate. Calls PUT /api/admin/reports/:id.

Backend Files Breakdown server/

Every file in the Express.js backend and what role it plays.

Core Server Files

index.js Entry

The server's main entry point. Loads environment variables via dotenv, initializes the database, imports all 9 route files, sets up CORS and JSON parsing middleware, mounts each route group under its /api/ prefix, adds a /api/health endpoint, and starts listening on port 3001.

db.js Database

Opens (or creates) the SQLite database file uniconnect.db. Enables WAL mode (Write-Ahead Logging) for better concurrent read/write performance, turns on foreign key enforcement, and runs CREATE TABLE IF NOT EXISTS for all 12 tables. Exports the db object that every route file uses to run queries.

.env Config

Environment variables file. Contains JWT_SECRET (the signing key for tokens) and PORT. Loaded by dotenv at startup. This file should never be committed to public repositories in a real production app.

seed.js / seed.cjs Script

Database seeding scripts. Run with npm run seed to populate the database with demo data — sample students (with Pakistani university names), admin accounts, study groups, resources, and sample conversations. Used for development and demonstrations.

Middleware — server/middleware/

auth.js Middleware

Exports two functions. authMiddleware: extracts the JWT from the Authorization: Bearer <token> header, verifies it against the secret key, and attaches the decoded payload ({ id, role }) to req.user. If invalid, responds with 401. adminMiddleware: checks if req.user.role === 'admin', otherwise responds 403. These are used as Express middleware — route files chain them before their handlers.

API Routes — server/routes/

auth.js Route → /api/auth/*

Handles registration (POST /register — validates @university.edu emails, hashes passwords with bcrypt, creates UUID), login (POST /login — verifies credentials, issues JWT), student verification (POST /verify), and password reset (POST /reset-password).

users.js Route → /api/users/*

Profile operations. GET /profile returns the logged-in user's full profile. PUT /profile updates name, bio, department, semester. PUT /privacy toggles visibility settings. GET /dashboard returns aggregate counts used by DashboardPage. All routes require authMiddleware.

partners.js Route → /api/partners/*

Study partner matching. GET / lists potential partners (excluding existing connections). GET /mine lists accepted partners. GET /requests lists pending requests. POST /request sends a partner request. PUT /:id accepts or declines. DELETE /:id removes a partner.

groups.js Route → /api/groups/*

Study group operations. POST / creates a group (auto-adds creator as member). GET /mine lists user's groups. GET /discover lists public groups. GET /:id returns full detail with members and announcements. POST /:id/join, DELETE /:id/leave, DELETE /:id/members/:userId (creator only), POST /:id/announcements (creator only).

resources.js Route → /api/resources/*

The largest route file (~8.5KB). Handles the complete resource lending lifecycle: listing, browsing (with search & category filters), borrow requests, approvals/rejections, handover scheduling, return confirmations, ratings, dispute filing, pause/resume toggling, and deletion. Maps to Use Cases 15-27.

messages.js Route → /api/messages/*

Messaging system. GET / lists all conversations with the last message preview. GET /:id/messages returns all messages in a conversation (each tagged with mine: 1/0). POST /send creates a message. If no conversation exists between two users, it automatically creates one.

notifications.js Route → /api/notifications/*

GET / returns all notifications for the current user. PUT /:id/read marks one as read. PUT /read-all marks all as read. Notifications are created by other route handlers (e.g., when someone sends a partner request, a notification is inserted).

lostfound.js Route → /api/lost-found/*

GET / returns all lost & found items with reporter names. POST / creates a new report (type: Lost/Found, description, location). PUT /:id/resolve marks an item as resolved/reunited.

admin.js Route → /api/admin/*

Admin-only endpoints (protected by both authMiddleware and adminMiddleware). GET /dashboard returns platform-wide aggregations. GET /users returns all accounts. PUT /users/:id/block toggles blocked status. GET /resources returns flagged listings. DELETE /resources/:id removes a listing. GET /reports returns all reports. PUT /reports/:id updates report status.

Configuration Files Root

The config files in the project root that control build tools, TypeScript, and linting.

package.json Config

The project's identity card. Lists the project name (sda-uniconnect), version, scripts (dev, build, lint, preview), all runtime dependencies, and all dev dependencies. npm install reads this file to know what to download.

vite.config.ts Config

Vite's configuration. Registers the React plugin (enables JSX/TSX compilation), and sets up the API proxy — any request to /api/* during development is forwarded to http://localhost:3001 where the Express server lives. This eliminates CORS issues.

tsconfig.json Config

The root TypeScript config. References two sub-configs: tsconfig.app.json (for source code) and tsconfig.node.json (for config files like vite.config.ts). This split ensures different TypeScript rules apply to browser code vs. Node.js code.

tsconfig.app.json Config

TypeScript settings for the React app. Key options: target: ES2020 (modern JS output), jsx: react-jsx (automatic JSX transform, no need to import React manually), strict: true (enables all type-checking), noUnusedLocals, and noUnusedParameters.

eslint.config.js Config

ESLint configuration. ESLint is a code quality tool that scans your code for potential errors, unused variables, and style issues. This config enables TypeScript-aware rules and React-specific hooks checks (e.g., "you forgot a dependency in useEffect").

index.html Entry HTML

The only HTML file. Contains a <div id="root"></div> where React mounts the entire app, and a <script type="module" src="/src/main.tsx"> tag that tells the browser to start loading the app. Vite uses this as the entry point for the build.

.gitignore Config

Tells Git which files to exclude from version control. Ignores node_modules/ (thousands of dependency files), dist/ (build output), and editor-specific files. These folders are regenerated from package.json so there's no need to track them.

📡 Backend Dependencies (server/package.json)

PackageWhat It Does
expressThe web server framework. Provides routing (app.get(), app.post()), middleware chaining, request/response handling, and the entire HTTP server infrastructure. The backbone of the backend.

Glossary of Terms Reference

Quick definitions for terms you'll encounter throughout the codebase.

TermDefinition
ComponentA reusable piece of UI in React. Can be as small as a button or as large as an entire page. Written as a JavaScript function that returns JSX.
JSX / TSXA syntax extension that lets you write HTML-like code inside JavaScript. <div className="card">Hello</div> is JSX. TSX is the same thing but in TypeScript files.
PropsShort for "properties." Data passed from a parent component to a child component. Like function arguments but for UI components. Example: <Avatar name="Ali" size={32} />
StateData that changes over time within a component. Managed with useState(). When state changes, React automatically re-renders the component to reflect the new data.
HookSpecial React functions that start with use. useState manages data, useEffect runs side effects (API calls, timers), useRef holds mutable references.
useEffectA hook that runs code after the component renders. Used for fetching data, setting up intervals, or subscribing to events. The dependency array controls when it re-runs.
API EndpointA specific URL on the server that performs an action. GET /api/users/profile fetches a profile, POST /api/auth/login logs you in.
RESTAn architectural style for APIs. Uses HTTP methods: GET (read), POST (create), PUT (update), DELETE (remove). Each URL represents a resource.
MiddlewareA function that runs between the incoming request and the route handler. Express middleware can check authentication, log requests, parse JSON bodies, etc.
CORSCross-Origin Resource Sharing. A browser security feature that blocks requests to different domains/ports unless the server explicitly allows it.
ProxyAn intermediary that forwards requests. Vite's proxy forwards /api requests to the Express server so the browser thinks everything comes from one origin.
HMRHot Module Replacement. When you save a file, Vite updates only the changed module in the browser without a full page reload. State is preserved.
Token (JWT)An encoded string containing user data (ID, role) signed with a secret key. Sent with every API request to prove identity. Expires after a set time.
Hash (bcrypt)A one-way transformation of a password. You can verify a password against its hash, but you can't reverse the hash back to the password.
Foreign KeyA column in one database table that references the primary key of another table. Enforces data integrity — e.g., a Message's senderId must refer to an existing User.
WAL ModeWrite-Ahead Logging. A SQLite performance optimization where writes go to a separate log file first, allowing simultaneous reads. Improves concurrency.
node_modulesThe folder containing all installed packages. Generated by npm install. Can contain thousands of files. Never committed to Git — regenerated from package.json.
